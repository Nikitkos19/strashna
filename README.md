# Задача

Мы решали задачу Авито по выделению дополнительных самостоятельных услуг из объявлений категории «Ремонт квартир и домов под ключ».

Нужно было по тексту объявления определить:

- `detectedMcIds` — какие дополнительные микрокатегории действительно присутствуют в объявлении
- `shouldSplit` — нужно ли создавать дополнительные черновики
- `drafts` — тексты предложенных черновиков по найденным услугам

# Идея

Объявления о ремонте под ключ часто содержат перечень отдельных услуг: сантехника, электрика, плитка, натяжные потолки, поклейка обоев и так далее.

Наша идея была в том, чтобы разделить задачу на три части:

- сначала определить, какие дополнительные услуги реально упоминаются в тексте
- затем понять, являются ли они самостоятельными и стоит ли дробить объявление
- после этого сгенерировать черновики для найденных микрокатегорий

# Подход

Мы используем гибридное решение из эвристик и простой ML-модели.

## 1. Детекция микрокатегорий

Для `detectedMcIds` используется rule-based анализатор.

Он опирается на:

- словарь микрокатегорий и ключевых фраз
- нормализацию текста
- стемминг словоформ
- поиск совпадений по фразам и их основам
- дополнительные эвристики для спорных категорий
- suppression-правила, чтобы уменьшать ложные срабатывания

Также мы добавили отдельные recall-синонимы для самых проблемных категорий:

- сантехника
- натяжные потолки
- укладка плитки
- штукатурные работы
- напольные покрытия

Это позволило заметно сократить число пропусков.

## 2. Решение о split

Для `shouldSplit` используется `LogisticRegression + TF-IDF`.

Модель обучается на текстах объявлений и предсказывает, нужно ли создавать дополнительные черновики.

Используем:

- word TF-IDF
- char TF-IDF
- `LogisticRegression`

Дополнительно мы откалибровали порог решения, чтобы уменьшить количество ложных split-срабатываний.

## 3. Генерация черновиков

Для `drafts` используется шаблонная генерация.

Для каждой найденной дополнительной микрокатегории мы формируем объект:

- `mcId`
- `mcTitle`
- `text`

Текст черновика строится на основе названия категории и найденных в объявлении ключевых фраз.

# Модель

Итоговое решение состоит из двух частей:

- rule-based детектор дополнительных микрокатегорий
- ML-классификатор `LogisticRegression` для `shouldSplit`

Такой подход оказался удачным, потому что:

- словарь и эвристики хорошо ловят конкретные услуги
- простая линейная модель быстро и стабильно решает задачу split
- решение остаётся понятным, интерпретируемым и быстрым

# Результаты

На датасете `rnc_dataset.csv` мы получили:

Решение хорошо находит дополнительные услуги и достаточно точно понимает, когда объявление действительно нужно дробить на отдельные черновики.

# Плюсы решения

- гибридный подход: сочетает эвристики и ML
- не требует тяжёлых моделей и дообучения LLM
- работает быстро и прозрачно
